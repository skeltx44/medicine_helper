## 서비스 요약

**약을 먹자**는 고령층을 위한 **AI 기반 복약 도우미 서비스**로,  
**약봉투 인식(OCR) · 복약 스케줄링 · 복약 기록 관리 · 복지 연계**까지  
한 번에 해결하는 **고령친화 스마트 케어 솔루션**이다.

- 멀티모달 비전 모델을 활용해 **약봉투 사진만 찍어도** 약 정보가 자동으로 정리되고  
- 고령층 전용 UX와 음성 안내 컨셉을 통해 **누구나 쉽게 복약을 관리**할 수 있도록 돕는다.

## 서비스 소개

### 서비스 개요

**‘약을 먹자’**는 고령층의

- 복약 실수  
- 복용 누락  
- 중복 복용  

문제를 해결하기 위해 개발된 AI 기반 복약 관리 서비스이다.

고령층은 작은 글씨의 약봉투, 복잡한 복용 지시, 기억력 저하로 인해
실제 생활에서 복약 오류를 겪기 쉽다. 이를 해소하기 위해 다음과 같은 기능을 제공한다.

- 📸 약봉지 촬영 → AI OCR로 약 정보 자동 추출  
- 📅 복약 스케줄 자동 생성 (일 단위 · 시간대 단위)  
- ⏰ 아침 / 점심 / 저녁 복약 알림 (설계)  
- 🎨 색 · 모양 · 크기로 재설명하는 고령친화 안내 문장  
- 📊 복약 기록 · 달력 기반 이행률 확인  
- 👨‍👩‍👧‍👦 보호자 알림 및 복지 연계 가능성  

**목표:**  
고령층이 “혼자서도” 쉽게 복약을 관리할 수 있는 환경을 만드는 것

---

### 타서비스와의 차별점

#### ① 고령층 전용 UX

- 큰 글씨, 고대비 색상, 단순한 버튼 구조
- 한 화면에 한 기능만 배치하는 “인지 부담 최소화” 설계
- 텍스트를 길게 읽기 어려운 사용자를 고려한 단순 문장 구성
- 음성 안내 기능을 고려한 UX 설계(콘셉트)

#### ② 멀티모달 AI 기반 약봉투 OCR

- **GPT-4o (Vision)**을 활용하여 단순 문자 인식을 넘어  
  ‘복용 횟수 · 기간 · 약 이름’의 의미 단위까지 자동 분류
- 한 봉투에 여러 약이 적혀 있어도  
  `medications[]` 구조로 여러 약을 한 번에 인식할 수 있게 설계
- `"500mg 정"`, `"250ML"`, `"알"` 등 용량/단위 텍스트는 후처리에서 정제

#### ③ 복약 설명을 “색 · 모양 · 크기”로 변환

노인이 이해하기 쉽게 약 이름 대신 외형 중심 설명으로 변환

예시:

"OO정 500mg"  
→ "작은 하얀색 둥근 약 1알"

의사, 약사, 보호자는 약 이름 기준으로,  
고령층 사용자는 색·모양·크기 기준으로  
서로 다른 표현을 공유할 수 있는 구조를 지향한다.

#### ④ 고령층 · 보호자 · 지자체를 모두 연결하는 구조

- 1차: 개인용 서비스 (노인 본인)
- 2차: 보호자 구독 모델
- 3차: 지자체 / 복지센터 확장 (독거노인 관리, 복지 사각지대 발굴)
- 4차: 병원 · 약국 연계 (처방·조제 정보 자동 연동)

단순 “알림 앱”이 아니라,  
개인–가족–지자체–의료기관까지 단계적으로 확장 가능한 구조를 목표로 한다.

#### ⑤ 인메모리 기반 프로토타입 → 확장 가능한 API 구조

현재는 해커톤용 프로토타입으로:

- 서버 메모리(In-memory) 기반 저장 구조
- 재시작 시 데이터 초기화

그러나 초기 설계에서부터:

- DB 연동
- API 확장
- 기관 연동(B2G / B2B)

을 고려해 RESTful API 구조를 잡았다.

## 구현 내용 및 결과물

### 1. 주요 기능

#### ① 약 봉투 인식 (OCR)

사용자가 앱/웹에서 약봉투를 촬영하면,  
프론트엔드가 사진을 base64로 인코딩하여 `/api/ocr`로 전송한다.

백엔드는 GPT-4o Vision을 호출해 다음 정보를 추출한다.

- `name` : 약 이름  
- `dosage` : 1일 복용 횟수 (정수)  
- `days` : 총 복용 일수 (정수)  

한 봉투에 여러 약이 적혀 있는 경우:

- `medications: [ { name, dosage, days }, ... ]` 형태로 여러 약 동시 인식

후처리:

- 용량 단위·불필요 텍스트 제거
- `dosage` 값에 따라 복용 시간대 계산
- 비정상 데이터(0, 너무 큰 값, 숫자+문자 혼합 등) 정리

#### ② 오늘의 약

서버에 저장된 `registered_date + days`를 기준으로  
“오늘이 복용 기간 안인지” 계산한다.

각 약의 `times` 필드(예: `["아침", "저녁"]`)에 따라  
아침 / 점심 / 저녁별 “오늘 먹어야 할 약 리스트”를 생성한다.

프론트엔드에서 시간대별 약 이름 리스트를 `/api/medications/convert`로 보내면,  
GPT-4o-mini가 약 이름 리스트를 `"작은 하얀색 둥근 약 1알"`과 같은 문장으로 변환한다.

사용자는 “오늘의 약” 화면에서:

- 시간대별 복약 안내 확인
- “복용 완료” 버튼 클릭 → `/api/medications/complete`로 기록 저장

#### ③ 복약 내역

`/api/history`를 통해 기간별 복약 기록을 조회한다.

- `start_date`, `end_date` (YYYY-MM-DD) 쿼리 파라미터

사용자는 특정 기간 동안:

- 언제, 어떤 약을, 어느 시간대에 복용했는지 확인 가능

진료 시:

- 의사에게 복약 이력 공유 → 복약 패턴 파악 및 약물 조정에 활용 가능

#### ④ 이달의 약

`/api/history/month` 엔드포인트로

- `year`, `month` 기준 해당 월 복약 데이터 조회

“이달의 약” 화면(thismonth.html)에서는:

- 달력 형태로 복약 이행률 시각화
- 복약한 날: `O`
- 복약 실패: `X`
- 하루 복약 횟수: 아이콘 개수로 표현

#### ⑤ 챗봇 기능

`/api/chat` + GPT-4o-mini 기반 고령층 친화 대화형 복약 가이드

예시 질의:

- “약 먹는 시간 알려줘”
- “이 약은 언제 먹어?”

시스템 프롬프트에서:

- 고령층 대상
- 짧고 쉬운 한국어 문장
- 마크다운 금지  

를 강제하여, 결과를 바로 UI 말풍선에 붙여도 읽기 쉬운 응답을 생성한다.

#### ⑥ 복약 알림 서비스 (설계 상태)

아침 / 점심 / 저녁 복약 시간 기반 푸시 또는 문자 알림 구성 설계

사용자 로직:

- 설정된 시간에 복약 알림 발송
- 30분 내 “복용 완료” 미응답 시, 본인 + 보호자에게 재알림 발송 구조

알림 트리거:

- `/api/notifications/start` : 알림 서비스 시작
- `/api/notifications/trigger` : 특정 시간대에 미복용 시 알림 발송

현재 해커톤 프로토타입에서는 알림 로직 및 API 설계 단계이며,  
이후 실제 SMS/푸시 연동 서비스를 붙일 수 있도록 구조화하였다.

## 앱 구조 개요 (Web + Android WebView)

이 프로젝트는 GitHub Pages에 배포된 웹페이지를 WebView로 감싼 Android 앱이다.

### 🌐 GitHub Pages (웹 프론트)

- HTML / CSS / JavaScript로 웹 프론트를 구현
- 이를 GitHub Pages를 통해 정적 웹사이트로 배포

서비스 URL:  
👉 https://skeltx44.github.io/medicine_helper/

이 주소로 접속하면 브라우저에서 동일한 서비스를 이용할 수 있다.

### 📲 Android WebView 앱 (webapp)

- Android 앱은 별도의 네이티브 UI를 많이 두지 않고,  
  WebView가 위 GitHub Pages URL을 로드하는 구조로 되어 있다.

앱 실행 → WebView 초기화  
WebView에서 `https://skeltx44.github.io/medicine_helper/`를 `loadUrl(...)`로 호출

결과적으로 웹 버전과 동일한 화면을 앱 형태로 제공한다.

즉, 브라우저를 열고 URL을 입력하는 과정을 생략하고,  
앱 아이콘 한 번으로 해당 웹 페이지에 바로 접근할 수 있는 전용 클라이언트 역할을 한다.

**✅ 요약**

- 프론트엔드(Web):  
  https://skeltx44.github.io/medicine_helper/ 에 호스팅된 웹페이지
- 클라이언트(App):  
  해당 페이지를 로드하는 Android WebView 앱

**장점:**

- 웹을 수정하면 앱 화면도 함께 반영
- 사용자는 네이티브 앱처럼 간편하게 접근 가능

## 구현 방식

### Frontend

#### 기술 스택

- HTML5
- CSS3
- Vanilla JavaScript

#### 레이아웃

- 모바일 기준 너비 390px에 맞춘 반응형 레이아웃
- 고령층을 위한 큰 폰트, 단순한 버튼, 높은 대비 색상 적용

#### 카메라 캡처

- `getUserMedia` 또는 `<input type="file" accept="image/*" capture>` 활용
- 촬영된 이미지를 base64로 변환하여 백엔드로 전송

#### UI/UX 설계 포인트

- 한 화면에는 한 가지 주요 기능만 보여주도록 설계
- 텍스트보다 아이콘, 큰 버튼 위주의 단순 구조
- 화면 전환 흐름:  
  로그인 → 메인 → (약 인식 / 오늘의 약 / 이달의 약 / 챗봇 등)

---

### Backend

#### 기술 스택

- Python 3
- Flask
- Flask-CORS (CORS 허용)

#### API 스타일

- JSON 기반 REST API
- `/api/...` 형태로 엔드포인트 구성

#### 데이터 저장 구조 (프로토타입)

- `medications_db: list[dict]`  
  - OCR 또는 수동 입력으로 등록된 약 정보 저장  
  - 예시 필드:  
    `id, name, dosage, days, times, registered_date, notification_times, image_base64` 등

- `medication_history_db: list[dict]`  
  - “복용 완료” 버튼 클릭 시 기록 저장  
  - 예시 필드:  
    `medication_id, time(아침/점심/저녁), completed_at, date`

#### 환경 변수

- `OPENAI_API_KEY`  
  - OpenAI API 호출에 사용  
  - 코드 내에 하드코딩하지 않고, 배포 환경(Render 등)의 환경변수로 설정  
  - 백엔드에서는 `os.getenv("OPENAI_API_KEY")`로 읽어 사용

---

### AI / OCR

#### GPT-4o (Vision)

- 약봉투 이미지 OCR 및 약 정보 추출
- 이미지 입력 → `medications: [ { name, dosage, days }, ... ]` JSON 구조 반환

#### GPT-4o-mini

- 노인 친화 복약 설명 생성
- 챗봇 응답 생성

#### 프롬프트 & 후처리

- Vision 모델이 정해진 JSON 스키마를 지키도록 프롬프트에서 강제
- “이 글자가 약 이름인지, 복용 횟수인지, 복용 기간인지”까지 해석하도록 유도

후처리 단계에서:

- 용량 문구 제거
- 비정상 값 정리
- `dosage`에 따라 시간대(`["아침", "저녁"]` 등) 계산
- 이름이 불명확한 약은 저장하지 않음 (추측 금지)

## 기능 명세서 & API 개요

아래는 주요 기능별로 정리한 API 개요이다.  
(실제 구현은 `/api/...` prefix를 갖는 엔드포인트 기준)

---

### 1) 약봉투 인식 & 약 정보 등록

#### 📌 약봉투 OCR – `POST /api/ocr`

**요청 예시**

    {
      "image": "data:image/jpeg;base64,..."
    }

또는 순수 base64 문자열만 전달해도 된다.

**동작**

- GPT-4o Vision 호출 → 약 이름, 1일 복용 횟수(`dosage`), 복용 일수(`days`) 추출
- 여러 약이 있을 경우 `medications[]` 배열로 반환
- 서버 내부 `medications_db`에 각 약 정보 저장

**응답**

- 첫 번째 약 정보 + 전체 `medications` 리스트

---

#### 📌 약 목록 조회 – `GET /api/medications`

- 현재까지 등록된 모든 약 정보를 배열 형태로 반환

---

#### 📌 약 등록(동기화) – `POST /api/medications`

- 프론트에서 수동 입력한 약을 서버와 동기화할 때 사용

요청 예시:

    {
      "name": "OO정 500mg",
      "dosage": 3,
      "days": 7
    }

- 서버에 새로운 약으로 저장

---

#### 📌 약 상세 조회 – `GET /api/medications/{id}`

- 특정 약 ID로 상세 정보 반환
- 존재하지 않는 ID는 에러 메시지 반환

---

### 2) 오늘의 약 & 노인 친화 설명

#### 📌 오늘 복용해야 할 약 조회 – `GET /api/medications/today`

- 각 약의 `registered_date`와 `days` 기준으로  
  **오늘 날짜가 복용 기간에 포함되는 약**만 필터링
- 약 한 개가 아침/저녁 두 번이면,  
  같은 ID가 `time`만 달리 여러 번 등장 가능

---

#### 📌 약 설명 변환 – `POST /api/medications/convert`

요청 예시:

    {
      "names": ["OO정 500mg", "다른약"],
      "time": "아침"
    }

**동작**

- GPT-4o-mini로 각 약을  
  `"작은 하얀색 둥근 약 1알"`처럼 **색/모양/크기**만으로 표현
- 입력된 약 개수만큼 줄 수를 맞추도록 후처리
- 마지막에 “을 물과 함께 드세요”와 같은 안내 문장 추가

**응답**

- 줄바꿈 기준으로 약별 설명이 포함된 문자열

---

#### 📌 복용 완료 기록 – `POST /api/medications/complete`

요청 예시:

    {
      "medication_id": 1,
      "time": "아침"
    }

**동작**

- 현재 날짜/시간 기준 복약 완료 기록을 `medication_history_db`에 저장
- 향후 알림 스케줄러와도 연동 가능

---

### 3) 복약 기록 · 복약 내역 · 이달의 약

#### 📌 복약 내역 조회(기간별) – `GET /api/history`

- 쿼리 파라미터:
  - `start_date` (YYYY-MM-DD)
  - `end_date` (YYYY-MM-DD)

**응답**

- 해당 기간 동안 기록된 복약 완료 정보 목록

---

#### 📌 이달의 복약 내역(달력용) – `GET /api/history/month`

- 쿼리 파라미터:
  - `year`
  - `month`

**응답**

- 해당 월의 복약 데이터를 날짜별로 그룹화  
- “이달의 약” 달력에서 O/X, 아이콘 개수 표현에 활용

---

### 4) 챗봇 기능

#### 📌 챗봇 – `POST /api/chat`

요청 예시:

    {
      "message": "약을 놓치면 어떻게 해요?"
    }

**동작**

- GPT-4o-mini 호출
- 시스템 프롬프트에서:
  - 고령층 대상
  - 쉬운 한국어
  - 마크다운 금지  
  를 강제

응답 예시:

    {
      "response": "약을 한 번 놓쳤다면 너무 걱정하지 말고, 다음 시간부터는 규칙적으로 드세요."
    }

- UI 말풍선에 바로 붙여 넣기 좋은 짧고 이해하기 쉬운 문장

---

#### 📌 헬스 체크 – `GET /api/health`

- 서버 동작 상태 확인용

응답 예시:

    {
      "status": "healthy"
    }

---

### 5) 복약 알림 서비스 (설계)

> 현재 해커톤 단계에서는 **알림 로직 & API 구조만 설계**된 상태이며,  
> 실제 SMS/푸시 연동은 이후 확장 단계에서 구현을 목표로 함.

#### 📌 알림 서비스 시작 – `POST /api/notifications/start`

- 사용자가 로그인하거나 알림 설정을 활성화할 때 사용
- (설계) 본인 및 보호자에게 “알림 서비스 시작” 문자 전송

#### 📌 복약 알림 트리거 – `POST /api/notifications/trigger`

- 특정 시간대에 복용 완료가 기록되지 않은 경우 호출
- (설계) 본인 + 보호자에게 복약 알림 전송
- 서버 스케줄러와 연동해 자동 호출 가능

## 향후 개선 혹은 발전 방안

### 1) 단계별 확장 모델

#### ① 1단계 – 무료 MVP 제공

노인 / 보호자 누구나 사용할 수 있는 완전 무료 서비스

**핵심 기능**

- 약봉투 촬영(OCR) · 자동 약 정보 인식
- 복약 스케줄 자동 생성
- 아침/점심/저녁 복약 알림(설계)
- 복약 기록 및 히스토리 확인

**목표**

- 실제 고령층 사용 환경에서 UI/UX 문제점 파악
- OCR 정확도 및 예외 케이스 수집
- 사용자 피드백 기반 서비스 고도화

---

#### ② 2단계 – 보호자 구독 모델

노인은 무료 유지, 보호자를 위한 프리미엄 기능을 월 소액 구독으로 제공

**주요 기능 예시**

- 보호자용 복약 성공/실패 알림
- 장기 패턴 분석 리포트 (복약 이행률, 누락 패턴 등)
- 여러 명의 부모님·가족을 한 계정에서 관리
- 간단한 약물 상호작용 안내 (기본 수준 정보 제공)

> 구독 모델을 통해 **안정적인 반복 수익(Recurring Revenue)**을 확보하고,  
> 동시에 보호자의 돌봄 부담을 줄이는 것을 목표로 한다.

---

#### ③ 3단계 – 지자체 · 복지기관 B2G 확장

지자체, 보건소, 노인복지관, 복지센터 등 B2G(Business to Government) 모델로 확장

공공기관 담당자를 위한 웹 대시보드 제공:

**기능 예시**

- 독거노인 복약 누락 탐지
- 고위험군 모니터링
- 방문 돌봄 우선순위 설정
- 위험 징후 조기 발견 (복약 누락 패턴 등)

> 공공기관 단위 계약을 통해 더 높은 단가와 안정적인 매출 확보가 가능하며,  
> 실제 복지 사각지대 해소에 직접 기여할 수 있다.

---

#### ④ 4단계 – 의료기관 연동

약국, 병원, 처방전 시스템과 연계하여 의료 생태계 전반으로 확장

**가능한 시나리오**

- 약국에서 조제 정보를 앱으로 자동 전송
- QR코드 기반 처방전 → 앱 복약 스케줄 자동 생성
- 병원 EMR 시스템과 연동하여 복약 이행률 기반 진료 지원
- 보험사와 연계해 복약 이행률이 높은 사용자에게 혜택 제공

> 장기적으로는 **“복약 데이터를 기반으로 한 헬스케어 플랫폼”**으로 발전 가능하다.
